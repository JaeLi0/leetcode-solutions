# LeetCode 教学 + 归档 Skill

## 概述

本 skill 将 LeetCode 刷题的两个环节合并为一个流程：
1. **教学**：按照 CLAUDE.md 中的教学规则，给出所有解法、标注推荐解、堵死错误方向
2. **归档**：用户触发后，生成 solution.py + notes.md 两个文件、更新专题链接、commit + push

每道题生成两个文件：
- **solution.py**：面试官看的——干净代码，推荐解在最上面，注释简洁
- **notes.md**：自己复习的——详细思路、逐行图解、易错点、关联题、追问洞察

---

## 完整工作流

### Step 1：识别题目

从用户输入中提取：
- 题号（如 42）
- 题目名称（如果用户没给，根据题号查）
- 英文 slug（用于文件命名，如 `trapping-rain-water`）
- 难度（Easy / Medium / Hard）
- 所属专题（two-pointers、sliding-window、hash-table 等）

如果用户只给了题号，根据已知信息填充其余字段。如果不确定 slug 或题名，问用户确认。

### Step 2：检查是否涉及新内容

在开始讲题之前，判断：

- **是否是用户第一次接触这个专题？** → 先讲专题通用套路，再讲具体题
- **是否涉及用户没见过的数据结构？** → 先给完整入门（概念、操作、心智模型、坑）
- **是否涉及新算法/方法？** → 先用 1-2 句说清核心思想

这些前置知识讲完后，再进入具体题目。

### Step 3：教学

严格按照 CLAUDE.md 中「教学规则」部分执行：

1. 给出所有解法（暴力 → 最优）
2. **解法之间必须有过渡**：讲清楚上一个解法慢在哪 / 差在哪，下一个解法怎么针对性解决
3. 每个解法包含：
   - 思路说明（中文）
   - 代码（Python，带中文注释）
   - 时间/空间复杂度 + **消耗来源说明**（不能只写 O(n)，要写为什么是 O(n)）
4. 标注推荐解、可复用模式、辅助理解标签
5. **推荐解必须做逐行图解**：
   - 先完整贴代码
   - 用最小例子（2-3 个元素），逐行代码 + 逐步状态图走一遍
   - 链表画链式图带指针，数组画状态带下标，递归画当前层输入输出
   - 纯数学/公式题、用户表示已理解时可跳过
6. Medium/Hard 题主动堵死错误方向
7. 列出易错点
8. 关联已做过的题（如果有）

**教学阶段不自动生成笔记文件。** 用户会多轮追问，等用户说「整理」「存一下」「下一题」时再触发归档。

### Step 4：归档（手动触发）

**触发条件**：用户说「整理」「存一下」「记一下」「归档」「下一题」「commit」「push」时触发。

**不要在第一轮教学结束后自动生成。** 用户会多轮追问，归档应包含整个讨论过程中的所有内容：解法讲解、追问中的补充说明、纠错、新的洞察。

**文件路径**：`solutions/{区间}/{题号4位}-{slug}/`

区间规则：
- 题号 1-100 → `0001-0100/`
- 题号 101-200 → `0101-0200/`
- 以此类推

#### 4a. 生成 solution.py（面试展示面）

```python
"""
{题号}. {英文题名} | {难度}
https://leetcode.cn/problems/{slug}/
Tags: {标签}
"""

# ⭐ 解法一：{推荐解名称}（推荐）
# 时间 O(?) — {消耗来源} | 空间 O(?) — {消耗来源}
class Solution:
    def methodName(self, ...):
        ...

# 解法二：{名称}
# 时间 O(?) — {消耗来源} | 空间 O(?) — {消耗来源}
class Solution2:
    def methodName(self, ...):
        ...

# 解法三：{名称}（暴力）
# 时间 O(?) — {消耗来源} | 空间 O(?) — {消耗来源}
class Solution3:
    def methodName(self, ...):
        ...
```

**solution.py 规则：**
- 推荐解永远放最上面，用 ⭐ 标注
- 每个解法一个 class（Solution, Solution2, Solution3...）
- 注释只写一行复杂度（含消耗来源），不写教学内容
- 关键步骤可加短注释，但不要大段解释
- 不出现「你可能会想」「暴力哪里慢」等教学语气
- 代码必须可直接提交到 LeetCode（Solution 类格式）

### Step 5：更新专题文件

打开 `topics/{专题}.md`，在题目列表中追加一行：

```markdown
| {题号} | [{中文题名}](../solutions/{区间}/{题号}-{slug}/) | {难度} | {关键词} |
```

如果专题文件不存在，用以下模板创建：

```markdown
# {专题名称}

## 核心思路

{该专题的通用套路，2-3 段}

## 题目列表

| 题号 | 题目 | 难度 | 关键词 |
|------|------|------|--------|
| {题号} | [{中文题名}](../solutions/{区间}/{题号}-{slug}/) | {难度} | {关键词} |
```

### Step 6：更新 README 进度表

更新根目录 `README.md` 中对应专题行的已完成计数和题号列表。

### Step 7：Git 提交

```bash
git add solutions/ topics/ README.md
git commit -m "feat(solutions): add LC {题号} {中文题名}"
git push
```

如果 push 失败，告知用户原因，提示手动处理。

## 特殊场景处理

### 用户说「下一题」

先归档当前题（如果还没归档），然后根据当前专题和已完成的题目，推荐下一道题。优先推荐 Hot 100 中同专题的未完成题目。

### 用户说「更新 LC xxx」（第二遍复习）

打开已有的 notes.md，更新内容（补充新理解、精炼思路、加新关联题），更新 `reviewed` 字段为当天日期。solution.py 一般不需要改，除非代码有更好的写法。

### 用户说「整理 LC xxx」但不需要教学

如果 `raw/{题号}.md` 存在，读取内容，按模板格式化后归档。
如果不存在，问用户是否需要先讲解。

### 用户要求重新讲某道已归档的题

正常教学，然后覆盖已有文件（旧版本保留在 git 历史中）。

### 用户粘贴了和 Claude（claude.ai）的对话内容

把对话内容当作原始素材，提取解法、思路、易错点，按模板重新整理后归档。

## 质量自检清单

归档前逐项确认：

**solution.py：**
- [ ] 推荐解在最上面，用 ⭐ 标注
- [ ] 每个解法都有复杂度 + 消耗来源
- [ ] 代码可直接提交 LeetCode（class Solution 格式）
- [ ] 无教学语气，注释简洁专业

**notes.md：**
- [ ] front matter 字段齐全（id, title, difficulty, tags, created, reviewed, recommended）
- [ ] 题号和目录名一致
- [ ] 解法之间有过渡说明（🔗）
- [ ] 推荐解有逐行图解
- [ ] 暴力解已包含
- [ ] Easy 题暴力标了「可跳过」
- [ ] Medium/Hard 有「暴力 → 优化链条」
- [ ] 易错点至少 1 条
